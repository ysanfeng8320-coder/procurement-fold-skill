# Procurement Fold — 总链路图（Skill 编排）

> 对齐「内容账号 skill 树」的写法：先有总目标与阶段，再往节点挂子能力。  
> 状态：编排规格（2026-07-12）。已有脚本按阶段挂载；未挂满的标 **缺口**。  
> 实现细节仍见 `SPEC.md` / `ARCHITECTURE_v2.md`；本文件是**入口与顺序**的真源。

---

## 0. 总目标（树根）

**一句话**：找到可赚钱的「产品 × 场景 × 价带 × 渠道」，并落到陕甘可打电话的动作。

不是：全国招标搜索引擎、万能预测中标、也不是自媒体内容流水线。

价值顺序（经验真值）：

1. 先锁利润带（例：装订机 8–40 千元）
2. 再确认场景拟合（份额 × 密度 × 客单，不是裸共现）
3. 再找中标集成商/服务商（货物/软件；排除物业服务）
4. 用名单行动；用验证层回写「有效边」

两条合法入口（都进同一条总链路，只是从不同阶段切入）：

| 入口 | 问法 | 先跳到 |
|------|------|--------|
| A. 物种优先 | 「公安局/法院这周推什么货」 | 阶段 2 → 4（`run_fold`） |
| B. 产品发现 | 「给我印控仪/装订机，找场景」 | 阶段 3（`scene_fit`）→ 4 |

旧硬规则「必须先给物种」**降级为入口 A 的约束**；入口 B 允许先给品。

## 2. 各阶段：子能力与现状

### 阶段 0 · 想清楚（一次性）

**产出**：一页销售假设（品、利润带、打法、区域、停止规则）

| 子能力 | 状态 | 备注 |
|--------|------|------|
| `$idea-court` | 可用（外部 skill） | 新想法先过约束 |
| 利润带声明 | **缺口**（口头有，未入库） | 例：装订机 8–40 千 |
| 打法选择：直客 / 供集成商 | **缺口** | 决定阶段4主交付是买方还是中标商 |

**停止规则示例**：利润带说不清 → 不准进阶段 3。

---

### 阶段 1 · 产品知识入库（不是「找对标」）

**为什么改名**：内容 skill 的「找对标」是因为还没有产品，要去外面找模仿对象。  
你这边**已有产品**，这一步是把内部资料结构化成「自家货 + 竞品货」知识，供后面采数与拟合使用。

**资料来源（投喂）**：

- 厂商培训、平时讨论、开会会议纪要  
- 产品介绍：PPT / Word（及同类文档）  
- 竞品：说明书、公开型号/价带、你做过替换引流的品牌（如天意兴隆）

**产出**：写入 product pack 的结构字段

| 字段 | 用途 |
|------|------|
| 型号与价带 | 阶段0利润带、阶段3客单筛 |
| 适用场景词 | 阶段3拟合先验 |
| 禁区/排除词 | **采集闸门**（物业、食堂…） |
| 竞品对照 | 阶段4话术与替换策略 |

| 子能力 | 状态 | 备注 |
|--------|------|------|
| `product_packs/*/raw/` 已有印控仪纪要示例 | 半有 | 未形成统一投喂→抽取流水线 |
| 资料投喂目录 + PPT/Word/纪要抽取 | **缺口** | 本模块主建设点 |
| 中标商频次（公开标讯） | 半有 | 属阶段4行动层，不替代本阶段资料入库 |

---

### 阶段 2 · 定位与包（半固化）

**产出**：可复用的 pack + 区域配置（吸收阶段1抽出的价带与排除词）

| 子能力 | 状态 | 路径 |
|--------|------|------|
| Product Pack | 有 | `product_packs/{seal_control,shredder,binding_machine,copier,signal_shield,rubber_track}/` |
| Species Pack | 有 | `species_packs/{court,procuratorate,police,agriculture}/` |
| 发现区 / 行动区 | 有 | 发现：皖浙京沪；行动：陕甘（可配） |
| 价带硬过滤写进 pack | **缺口** | 来自阶段1 |
| 采集排除词写进 pack | **缺口** | 物业/食堂等，供闸门使用 |

命令（入口 A 主路径）：

```bash
python3 skills/procurement-fold/scripts/run_fold.py \
  --species court \
  --portfolio binding_machine,seal_control \
  --discovery 安徽,浙江,北京,上海 \
  --action 陕西,甘肃 \
  --reuse-local
```

### 采集闸门 · 阶段 3 之前必过

场景发现要**大量好样本**才稳（公安等稀缺场景可能要千级甚至约 5000 条硬投入；API 按条收费）。  
当前印控仪/矩阵多为**小样本测流程**，不能当成大样本结论。

**不过闸，不准把预算打满。**

| 闸门项 | 规则 |
|--------|------|
| 品类边界 | 办公货物/软件向；排除物业、保洁、食堂、维保、劳务等 |
| 关键词 | 来自阶段1 pack；禁止易混词抬相关 |
| 预算 | 测试额度（如≤200）与正式上限（如≤5000）分开写进 run |
| 均分 | 多关键词均分额度，禁止一词吃光 |
| 本地优先 | 先吃 `scout_data` / 已有 fold_runs，再决定是否加码 API |

| 子能力 | 状态 |
|--------|------|
| `scene_fit` 服务类标题排除 | 半有 |
| run 级闸门检查清单 + 硬拦截 | **缺口** |

### 阶段 3a · 数据分析（必须先锚定系统客户）

**对，场景发现不能直接上。** 顺序是：数据到位 → **数据分析** → 寻找规律 → 场景发现。

**硬规则（相对早期标注的改正）**：

| 旧做法 | 新做法 |
|--------|--------|
| 把采购方揉成 a/b/c/d/e/f/g 一类扁标签 | **定点物种**：公安局就是公安局，检察院就是检察院 |
| 采购方式/品类标签经常变，却拿扁标签当底座 | 公告类型可变；**系统客户身份尽量钉死** |
| 全库一锅分析再猜场景 | 先定所属系统，再在切片上分析 |

每条结构表至少含：`species_id`（系统客户）+ 产品命中 + 价带 + 年月 + 买方/中标商。

| 子能力 | 状态 |
|--------|------|
| Species Pack + `match_species` | 有 |
| 结构流水线 `structure.py` | 有 |
| 标注质检（串物种、集成商冒充买方） | **缺口** |

### 阶段 3b · 寻找规律（物种切片上运算）

在「公安局切片 / 检察院切片 …」上做统计与图推理（默认不上神经网络）：月度涌现、价带、跟风、关联缺口，产出可检验规律卡片。

| 子能力 | 状态 |
|--------|------|
| `fold_rules` / portfolio 机会分 | 有 |
| 规律库回写（有效/弱/否决） | **缺口** |

### 阶段 3c · 场景发现（规律够用之后）

**产出**：场景拟合卡（系统排行 + 份额/密度/客单）

没有 3a 物种锚定 + 3b 规律，所谓发现只是标题相关，不可信。

| 子能力 | 状态 | 路径 |
|--------|------|------|
| `scene_fit` | 有（印控仪小样本） | `kg_engine/fold/scene_fit.py` |
| 装订机对照回测 | **缺口** | 用已知真值校准 |
| 稀缺系统大样本 | **缺口** | 过闸后的硬投入 |

```bash
cd kg_engine && PYTHONPATH=. python3 -B fold/scene_fit.py seal_control
```

**合格边**：高客单 + 够密 + 份额不虚 + 可行动。  
**淘汰边**：复印纸式共现、过稀渗透、纯服务类。  
**花费原则**：小样本证明方法 → 闸门通过 → 再为高价值假设加码。

### 阶段 5 · 电话补全（必做）→ 人行动

**硬规则**：名单产出后必须跑电话补全；禁止只给单位名就结束。

工具：`kg_engine/fold/phone_complete.py`（封装 `phone_enricher`）

```bash
# 只扫本地公告入库
cd kg_engine && PYTHONPATH=. python3 -B fold/phone_complete.py --harvest-only

# 对某次 run 补全（默认不调 API）
PYTHONPATH=. python3 -B fold/phone_complete.py --run ../fold_runs/<id>

# 小额度 API 补号
PYTHONPATH=. python3 -B fold/phone_complete.py --run ../fold_runs/<id> --api-budget 30 --province 陕西

# 跑 fold 时一并补全
python3 skills/procurement-fold/scripts/run_fold.py ... --phone-complete --phone-api-budget 20
```

策略顺序：

1. scout_data 正文/字段抠号 → `contacts.json`  
2. 买方本人号  
3. 买方/参照单位历史中标商  
4. 物种渠道池里**已有真号**的集成商（优先）  
5. 可选 API（世舶/知了）小预算  

产出：`leads.json` 更新 + `phone_report.html`（直联 / 经集成商联系：XX☎…）

| 子能力 | 状态 |
|--------|------|
| 本地 harvest + 经集成商路径 | **已有** |
| API 补号 | 有（额度/命中率视上游） |
| 人外呼 | 人做 |
| CRM 回写 | **缺口**（可选） |

## 3. Agent 怎么跑这条树

触发词：`procurement-fold` / `采购折叠` / `场景拟合` / `货盘机会`

1. 先问用户卡在哪一阶段（0–6）；不明确则默认从阶段 0/3 判断。  
2. 只跑当前阶段及直接下游；禁止一上来全库重训。  
3. HTML 报告用系统浏览器打开（`open path/report.html`）。  
4. 新想法 / 扩品 / 扩省 → 先 `$idea-court`（阶段 0），再改 pack。  
5. 知了网额度不足 → 本地 `scout_data` + 已有 `fold_runs`，并在报告注明。

## 6. 现在不要做什么

- 不要把内容账号的「找对标」照搬成「再找一个产品」  
- 不要未过采集闸门就打满 API 额度（物业等噪声按条收费）  
- 不要用小样本场景拟合冒充大样本结论  
- 不要先做全品类万能知识图谱平台  
- 不要在阶段 3 未过筛时大规模外呼  
- 不要用「相关就推荐」替代利润×密度拟合  
- 不要跳过回写只堆新 HTML 报表
