# Procurement Fold v2 — 架构草案（物种优先）

> 状态：**仅架构，未改代码。** 确认后再改 Skill / Spec / 实现。  
> 日期：2026-07-11  
> 相对 v1 的核心转向：从「产品搜客户」改为「物种定客户 → 采购序列 → 货盘机会」。

---

## 1. 一句话

先锁定**客户物种**（如公安局、检察院），在稳定物种上建立**采购事件序列**与**跨地跟风**；再对你代理的**多个产品**分别给出「这类客户买它的相对可能性 + 依据」。

提问时**必须**提供物种颗粒度；产品是货盘上的多个问题，不是唯一入口。

---

## 2. 为什么改

| v1（产品找客户） | 问题 |
|------------------|------|
| 搜印控仪 → 杂单位 | 「买方系统」扁标签抹平公安/农业/人社差异 |
| 从标题反推客户是谁 | 标注不稳，证据易脏 |
| 一次只服务一个本品 | 你同时代理装订机、打码机时要重跑整套 |

| v2（物种找需求） | 收益 |
|------------------|------|
| 先定公安局/检察院 | 物种属性稳定，可纵向建序列 |
| 序列上标「买过什么」 | 预测下一缺口，而不是猜标题 |
| 横向看甲乙丙地同类 | 兄弟单位抄作业可销售话术化 |
| 货盘多产品打分 | 一次物种，多产品机会对照 |

## 3. 核心对象

### 3.1 Species Pack（物种包）— 必选

定义「我是谁家的采购视角」。

```text
species_packs/<species_id>/
  species.yaml      # 物种名、匹配规则、层级、排除规则
  axes.yaml         # 稳定属性轴（层级/条线/场景），不是扁「系统」枚举
  industry_notes.md # 行业信息（可选：政策、牵头部门、采购习惯）
  sources.md
```

示例物种（第一版只做一个楔子）：

- `police` 公安局  
- `procuratorate` 检察院  
- `court` 法院（可选）

`species.yaml` 要点：

- **匹配**：单位名/标题含哪些词算本物种（公安局、公安分局…）  
- **排除**：派出所是否计入主序列（建议 v2.0 主序列不含派出所，或单列 `police_local`）  
- **层级**：省厅 / 市局 / 分局（分局可标为下级，不与市局混为同一实体）  
- **地域角色**：发现区 vs 落地区（沿用双层地理）

### 3.2 Product Pack（产品包）— 可多个

沿用现有 `product_packs/`（印控仪、装订机、打码机各一份）。

每个产品包仍含：词典、约束、产品说明。  
在 v2 里它的角色是：**货盘上的一个评分对象**，不是搜客户的入口。

### 3.3 采购事件（Event）— 序列里的「氨基酸」

一条近 N 年、标题真实命中某产品族的公告/中标，挂在**具体单位实体**上：

```text
Event:
  unit_id / unit_name     # 具体公安局，不吞成标签
  species_id              # police
  region / level          # 陕西 / 市级
  product_family          # 由哪个 product pack 命中
  hit_tier                # target|near|related（相对该产品包）
  date
  amount_bucket（可选）
  source_id / title
```

**序列** = 同一 `unit` 上按时间排序的 Event 列表。  
**物种画像** = 该 species 下所有单位序列的统计汇总 + 跨地扩散模式。

### 3.4 货盘（Portfolio）

一次提问携带的产品包列表：

```text
portfolio: [seal_control, binding_machine, coding_machine]
```

## 4. 提问契约（Agent / CLI 输入）

**必填：**

1. `species` — 物种包 id（如 `police`）  
2. `action_regions` — 落地省（如陕、甘）  
3. `portfolio` — 至少一个产品包  

**选填：**

4. `discovery_regions` — 学规律的省（默认皖浙京沪）  
5. `max_age_years` — 默认 3  
6. `api_budget` — 上限，不强制用满  

非法提问（应拒绝或追问）：

- 只给产品、不给物种  
- 物种颗粒度大于「局/院」一级却未定义 pack（如只说「政府」）

---

## 5. 流水线（四段，语义重定义）

```text
1. Scope（定界）
   加载 Species Pack + Portfolio
   确定发现区 / 落地区 / 近 N 年窗

2. Ingest（接入）
   按「物种关键词」拉/复用标讯（主）
   再在结果上用各 Product Pack 词典打产品命中（辅）
   禁止：因文件名把无关标题抬成 related
   禁止：超龄证据进入序列

3. Structure（结构）
   单位实体消歧（尽量：同市「XX市公安局」合并）
   打上物种轴（层级等）
   生成 Event，按单位排成序列
   不做「万能买方系统」扁标签

4. Fold（折叠）
   纵向：物种内采购共现、先后顺序、缺口
   横向：发现区单位买了 P → 落地区同物种未买 P 的扩散名单
   对 portfolio 每个产品输出机会分

5. Act（行动）
   销售版报告：按产品对照 + 落地区单位名单 + 证据（近 N 年）
```

（实现上 Fold/Act 可合并为一步；逻辑上分开。）

## 6. 折叠逻辑（v2.0 统计，不上神经网络）

### 6.1 纵向 — 物种采购画像

在发现区 + 落地区的该物种单位上统计：

- 各产品族出现频次 / 单位覆盖率  
- 常见先后顺序（先电子印章，后印控硬件…）  
- 「已有 A 未有 B」的条件缺口率  

### 6.2 横向 — 跨地跟风

若发现区某物种单位在窗口内出现产品 P，  
则落地区同物种、同级或下级、近 N 年未见 P 的单位 → 扩散候选。

### 6.3 货盘打分（相对可能性）

对每个 `product_pack` 输出 0–100 的**相对分**（非校准概率），建议因子：

| 因子 | 含义 |
|------|------|
| 物种覆盖率 | 该类单位里出现过本品的比例 |
| 缺口率 | 已有关联品、未见本品 |
| 近期涌现 | 1–2 月窗口是否抬头 |
| 横向扩散 | 发现区已现、落地区未现的密度 |
| 先验约束 | Species/Product 纪要约束加权 |

报告中写「相对高/中/低 + 依据」，可附分数；避免「72.3% 会买」这种假精确，除非以后单独校准。

## 7. 输出（销售优先）

### 7.1 主报告 `report.md`（老板/销售）

1. 物种与范围（公安局 · 陕甘 · 近 3 年）  
2. 一句话：本物种当前最值得推的货盘排序  
3. 货盘对照表：产品 | 相对机会 | 主要依据 | 建议动作  
4. 关键发现：纵向序列规律 + 横向跟风  
5. 落地区名单：按产品分列或总表（单位、缺口、证据日期、命中词）  
6. 卡点：电话/渠道  

### 7.2 技术明细 `report_tech.md`

Event 统计、序列样例、分数因子明细。

### 7.3 机器可读

`events.jsonl` / `unit_sequences.json` / `portfolio_scores.json` / `leads.json`

| 模块 | v2 角色 |
|------|---------|
| `product_packs/*` | 保留；变为货盘元素 |

## 10. 最小楔子（确认架构后的实现顺序）

1. 只做物种：`police`（公安局）  
2. 货盘先只挂：`seal_control`（印控仪）— 验证物种路径  
3. 再挂第二个产品包做「一物种多产品」冒烟  
4. 仍：近 3 年、标题真实命中、销售版报告、默认不上模型  

成功标准：

- 不给物种会追问/拒绝  
- 公安序列可读，证据不出现钢琴类误伤  
- 货盘表能回答「印控仪 vs 下一产品谁更值得推」  

---

## 11. 明确不做（本架构版）

- 不把「所有局」收成一个买方系统枚举  
- 不上神经网络概率模型  
- 不一次定义全国所有物种  
- 不以产品搜索为唯一主入口（产品只作货盘）  
- 不输出无物种、无近窗、无标题命中的名单
